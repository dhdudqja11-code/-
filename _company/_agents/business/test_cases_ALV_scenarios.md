# 📘 테스트 케이스 매트릭스: ALV 시뮬레이션 기반 (Test Cases for RiskAssessmentGateway)

## 목적
개발된 `RiskAssessmentGateway`가 최소한 다음 세 가지 고위험 시나리오를 감지하고, 논리적인 재무적 손실액(ALV) 보고서를 출력하는 것을 검증합니다.

## 📝 테스트 케이스 상세 (Test Case Detail)

### TC-A: PII 데이터 유출 (규제 리스크)
*   **입력 조건:** 시스템 로그에 정상적인 사용자 활동과 함께, 평소와 다른 대규모의 `SELECT * FROM users` 쿼리 및 다운로드 트랜잭션이 기록됨. 이 트랜잭션은 기존 보안 모듈을 우회했거나 감지되지 않았다고 가정함.
*   **기대 결과:**
    1.  위험 레벨: **Critical (최상)** 플래그 발생.
    2.  감지 근거: 비정상적인 데이터 접근 규모와 출처 불명확성.
    3.  출력 보고서 내용: 위 섹션 1의 ALV 구조를 기반으로, 손실액 추정치($3M+) 및 법적 책임에 대한 경고 메시지를 포함해야 함.

### TC-B: 데이터 Provenance 조작 (지식재산권 리스크)
*   **입력 조건:** 특정 모델 학습 요청이 들어왔으나, 해당 트랜잭션의 원본 데이터셋(`SourceDataId`)이 체인 해싱된 감사 로그에 기록되어 있지 않거나, **최근 24시간 내에 삭제된 흔적**이 포착됨.
*   **기대 결과:**
    1.  위험 레벨: **High (높음)** 플래그 발생.
    2.  감지 근거: 데이터의 '출처 증명(Provenance)' 체인 끊김 감지.
    3.  출력 보고서 내용: 위 섹션 2의 ALV 구조를 기반으로, IP 침해 소송 리스크($2M+) 및 데이터 무결성 복구 필요성을 강조해야 함.

### TC-C: 내부자 트랜잭션 변조 (무결성 리스크)
*   **입력 조건:** 정상적인 운영 시간대에, 시스템의 관리자가 마치 백도어처럼 특정 핵심 재무 트랜잭션(예: 수수료 계산 로직)의 로그 기록을 **직접 삭제하거나 값을 수정**한 시도가 감지됨.
*   **기대 결과:**
    1.  위험 레벨: **Critical (최상)** 플래그 발생.
    2.  감지 근거: 체이닝 해싱 기반 감사 블록(`AuditBlock`)의 무결성 파괴 시도 포착.
    3.  출력 보고서 내용: 위 섹션 3의 ALV 구조를 기반으로, 법적 증명력 상실($2.5M+) 및 **Immutable Audit Gateway(IAG)**의 필요성을 강조해야 함.
