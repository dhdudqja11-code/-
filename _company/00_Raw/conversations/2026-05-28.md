### 📜 원격 제어 프로그램 MVP 개발 계획서: 법적 무결성 게이트웨이 (AuthGateway) 구축

#### 1. 아키텍처 목표 및 핵심 원칙
*   **최우선 목표:** 모든 접근 시도에 대해 **불변의 감사 기록(Immutable Audit Log)**을 생성하는 것. 단순히 '로그인 성공' 여부를 넘어, 누가(Who), 언제(When), 어떤 리소스(What)에, 어떤 권한(Scope)으로 접근했는지를 모두 증명해야 합니다.
*   **기술 스택 방향성:** 서버 측에서 처리되어야 하며, 세션 관리보다는 **JWT (JSON Web Token)** 기반의 Claim-based 인증을 채택하여 Stateless하면서도 강력한 권한 검증이 가능하도록 설계합니다.
*   **보안 요구사항:** OAuth 2.0 Client Credentials Grant Type 및 Refresh Token Rotation을 기본으로 적용해야 합니다.

#### 2. 핵심 컴포넌트 정의: AuthGateway API 명세
우리는 이 모듈을 `AuthGateway`로 명명하고, 모든 요청은 반드시 이곳을 거치도록 강제합니다.

| 엔드포인트 | HTTP 메서드 | 기능 설명 | 필수 파라미터 (Body) | 응답 구조 예시 | 보안 고려 사항 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `/v1/auth/login` | `POST` | 사용자 인증 및 최초 토큰 발급. | `username`, `password`, `mfa_code` (선택) | `{ "access_token": "...", "refresh_token": "..." }` | **Argon2** 해싱 필수, MFA 강제화 고려. |

1.  **[유효성 검사 실패] 잘못된 자격 증명 시도:**
    *   **테스트 내용:** 존재하지 않는 사용자 ID와 비밀번호를 조합하여 `/v1/auth/login`을 호출했을 때, 401 Unauthorized가 반환되며, 시스템 로그에 **'Credential Failure Attempt'** 기록이 남는지 검증. (사용자에게는 실패했다는 정보 외의 디테일은 노출 금지.)
2.  **[인가 정책 우회] 권한 없는 리소스 접근 시도:**

3.  **[토큰 무결성] 만료된 Refresh Token 사용 시도:**
    *   **테스트 내용:** 이미 서버에서 블랙리스트에 등록되었거나 유효기간이 지난 `refresh_token`을 가지고 `/v1/auth/refresh`를 호출했을 때, 즉시 403 Forbidden이 반환되고, 해당 토큰의 **'Invalidated'** 기록이 남아 있는지 검증.
4.  **[외부 입력 공격] SQL Injection 시도:**
    *   **테스트 내용:** 사용자 이름이나 비밀번호 필드에 `'; DROP TABLE users; --`와 같은 악성 문자열을 삽입하여 `/v1/auth/login`을 호출했을 때, DB 쿼리가 실행되지 않고 파이썬 레벨에서 유효성 검사(Validation)를 통과하지 못하는지 확인.
5.  **[Rate Limiting] 초당 요청 과다 시도:**
    *   **테스트 내용:** 단일 IP 주소에서 짧은 시간 내에 10회 이상 `/v1/auth/login`을 호출했을 때, 시스템이 자동으로 **'Too Many Requests' (429)**를 반환하며, 해당 IP를 임시 블랙리스트(Rate Limit)에 추가하는지 검증.

### 1. 개요 및 목표 설정
본 문서는 원격 접속 프로그램의 핵심 모듈인 `AuthGateway`에 적용해야 할 최고 수준의 인증 및 권한 관리 기준을 제시합니다. 목적은 단순 접근 제어를 넘어, **'모든 트랜잭션에 대한 법적 무결성 증명(Immutable Proof)'**을 확보하는 것입니다.

### 2. 핵심 기술 표준 분석 (Latest Encryption Protocols)

| 프로토콜/개념 | 최신 동향 및 중요 변경점 | 적용 원칙 및 이점 (Why?) |
| :--- | :--- | :--- |
| **OAuth 2.1 & Bearer Token** | OAuth 2.0의 취약점을 보완하여 'Bearer Token' 사용 시 토큰 폐기(Token Revocation) 과정을 의무화함. 특히, Client Credentials Flow 사용 시 `PKCE (Proof Key for Code Exchange)` 적용을 강력히 권장하며, 이는 중간자 공격(MITM) 방어에 필수적입니다. | **[취약점 대응]** 단순 토큰 유출만으로는 접근 불가하게 만듦. 모든 요청마다 클라이언트의 출처 증명을 강제하여 보안성을 높임. |
| **Zero Trust Architecture (ZTA)** | "절대 신뢰하지 않고, 항상 검증한다(Never Trust, Always Verify)"가 핵심입니다. '경계 방어' 개념을 폐기하고, 모든 사용자/장치/API 호출마다 최소 권한 원칙(Least Privilege)에 입각한 동적 정책 평가를 요구합니다. | **[권한 확장]** 네트워크 위치나 IP 주소와 무관하게, 요청의 맥락(Context), 사용자의 역할(Role), 장치의 건강 상태(Device Health) 등 다차원적인 요소를 기반으로 접근을 판단하여 Lateral Movement 공격을 원천 차단합니다. |
| **실시간 API 권한 검증 (Policy Enforcement)** | 정적(Static) ACL(Access Control List) 방식에서 벗어나, 요청 시점의 비즈니스 로직 및 규제 준수 여부를 실시간으로 평가하는 **정책 기반 접근 제어(Policy-Based Access Control, PBAC)** 시스템이 필수입니다. | **[법적 무결성]** 단순한 '읽기/쓰기' 권한을 넘어, "사용자 A가 B 데이터에 대해 C 목적일 때만 트랜잭션 가능"과 같이 비즈니스 규칙까지 강제하여 법규 위반 시도를 사전에 차단합니다. |

class AuthGateway:
    """
    [System Under Test] 원격 제어 프로그램의 인증 및 권한 게이트웨이 (v1.0 Mock)
    실제 서비스 로직은 이 클래스의 메서드 내부에서 구현되어야 합니다.
    테스트를 위해 핵심 비즈니스 로직과 분리된 형태로 설계합니다.
    """

    def __init__(self, token: str):
        self.token = token
        self._is_authenticated = self._validate_token(token)

    @staticmethod
    def _validate_token(token: str) -> bool:
        # 실제로는 JWT 파싱 및 만료/서명 검증 로직이 들어갑니다.
        return "valid_" in token and len(token) > 10

    def get_user_role(self) -> str | None:
        if not self._is_authenticated:
            raise PermissionError("Authentication Failed: Invalid or missing token.")
        # 토큰에 포함된 사용자 역할 정보를 파싱한다고 가정합니다.
        return "admin" if "super" in self.token else ("analyst" if "valid_" in self.token else None)

    def access_resource(self, resource_id: str, required_role: str = "guest") -> dict:
        """
        특정 리소스에 접근하는 핵심 로직. RBAC 검증이 필수입니다.
        """
        current_role = self.get_user_role()

        if not current_role:
            raise PermissionError("Authentication Failed: Token could not be verified.")

        # 1. 권한 우회(RBAC) 체크 (가장 중요!)
        if required_role != "guest" and current_role != required_role and current_role != "admin":
             raise PermissionError(f"Authorization Denied: User role '{current_role}' lacks required permission '{required_role}'.")

        # 2. 필수 입력값 체크 (Input Validation)
        if not resource_id or not isinstance(resource_id, str):
            raise ValueError("API Input Error: Resource ID must be a non-empty string.")

        return {
            "status": "success",
            "data": f"Successfully accessed resource {resource_id} with role {current_role}.",
            "audit_log": f"Audit logged for access attempt by user {current_role}"
        }
# End of AuthGateway Mock Definition
