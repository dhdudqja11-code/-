아래에 서비스 레이어 인터페이스 스펙(API Specs), 데이터 흐름도(Data Flow Diagram)의 구조 정의, 그리고 예외 처리 체크리스트를 상세히 정리했습니다. 개발팀에게 전달하여 아키텍처 검토를 요청해주세요. ✅

### 📋 IAG 게이트웨이 설계 가이드라인 (V1.0)

#### 1. 핵심 데이터 흐름도(Data Flow Diagram: 구조 정의)
모든 API 요청은 아래의 **5단계 강제 처리 경로**를 거쳐야 합니다. 이 순서를 벗어나면 절대 안 됩니다.

*   **[Start] 사용자 요청 (Request)** $\rightarrow$
*   **① 인증 게이트웨이 (Auth Service):** JWT 검증 및 권한 확인. (Token/User ID 추출) $\rightarrow$
*   **② 감사 블록 생성 준비:** 트랜잭션 시작 시점 기록 (`timestamp`, `user_id`, `action`). $\rightarrow$
*   **③ 비즈니스 로직 서비스 (Business Logic):** 실제 핵심 기능 실행 및 위험 예측. (가장 중요한 데이터 처리) $\rightarrow$
*   **④ 감사 블록 완성:** 최종 결과, 성공/실패 여부, 그리고 트랜잭션 전 과정을 `AuditBlock`으로 기록하고 DB에 저장. $\rightarrow$
*   **[End] 응답 (Response)**

#### 2. 필수 인터페이스(API Specs) 정의 및 구조화
| 서비스명 | 역할 | 주요 함수 / 엔드포인트 | 입력 데이터 (Input) | 출력 데이터 (Output) | 핵심 체크 포인트 |
| :---: | :---: | :---: | :---: | :---: | :---: |
| **`AuthService`** | 인증 및 권한 검증 | `get_user(token)` | JWT Token | `UserPayload` (ID, Role) / Error | 토큰 만료/위조 시 즉시 401 에러 반환. |
| **`GatewayCore`** | 게이트웨이 관리자 | `run_transaction()` | `payload`, `data` | `{result: ..., audit_id: ...}` | 모든 호출을 감싸고, 시작과 끝에 AuditBlock 강제 삽입. |
| **`BusinessLogicService`** | 핵심 비즈니스 로직 | `simulate_risk(user_data)` | 사용자 데이터 (JSON) | 분석 결과 (`RiskScore`, `LossAmount`) | 예측 모델의 버전 및 사용된 파라미터를 AuditBlock에 기록해야 함. |

#### 3. 법적 증명력 확보를 위한 예외 처리 체크리스트 (⚠️필수 점검 사항)
단순히 '500 Internal Server Error'로 끝내면 안 됩니다. 모든 오류는 **"무슨 일이 일어났는지, 왜 실패했는지, 누가 시도했는지"**가 명시되어야 합니다.

*   **🔑 인증 관련 예외:**
    *   [ ] 토큰 만료 (Expired Token): 에러 메시지에 만료 시간 명시.
    *   [ ] 권한 위반 (Permission Denied): 해당 API 접근에 필요한 최소 권한(Role)을 구체적으로 명시.
*   **💾 데이터 무결성 예외:**
    *   [ ] 입력값 유효성 검사 실패: 어떤 필드가 어떤 규칙(예: 숫자여야 함, 10글자 이상 등)을 위반했는지 명확히 제시.
    *   [ ] 외부 시스템 연결 오류 (DB/API): 에러 발생 시도한 외부 시스템 이름과 마지막 호출 파라미터를 AuditLog에 기록.
*   **⚙️ 비즈니스 로직 예외:**
    *   [ ] 필수 데이터 누락: 서비스 실행에 필요한 핵심 데이터가 빠졌을 경우, 해당 데이터를 명시하고 트랜잭션을 중단. (이유: 분석 불가)
    *   [ ] 시스템 버전 불일치: 사용된 알고리즘/모델의 버전을 기록하여 나중에 재현 가능하도록 보장.
