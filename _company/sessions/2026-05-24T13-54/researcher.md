# 🛡️ Mini ROI 시스템: 운영 환경 규제 준수 검증 체크리스트 (V1.0)
**대상 모듈:** API Gateway 및 인증/권한 관리(AuthN/AuthZ) 서비스
**검토 목적:** 법적 책임 회피(Avoided Loss) 관점의 방어력 확보 입증

## Ⅰ. 데이터 무결성 및 감사 추적 (Auditability & Immutability) - [Critical]
이 섹션은 시스템에서 발생하는 모든 중요한 트랜잭션을 '불변의 기록'으로 남기는 데 초점을 맞춥니다. 단순한 DB 로그 저장을 넘어, 블록체인 기반의 원장(Ledger) 개념을 시뮬레이션해야 합니다.

| No. | 검증 항목 (Verification Point) | 테스트 방법 (Test Case/Scenario) | 기대 결과 (Expected Compliance Output) | 관련 규제 준수 요소 |
| :---: | :---: | :---: | :---: | :---: |
| **1.1** | **트랜잭션 원장 기록 의무화** | 모든 `write_data` 요청(데이터 변경 시도)에 대해 롤백 여부와 관계없이 트랜잭션을 '원장에 커밋'하는지 테스트. | ① **Immutable Hash 생성:** 해당 트랜잭션의 데이터 내용, 타임스탬프, 사용자를 포함한 SHA-256 해시 값을 생성하고 이를 로그 체인에 연결해야 함. ② **이전/이후 값 명시:** 변경된 필드명과 *변경 전 값(Old Value)*, *변경 후 값(New Value)*을 필수적으로 기록함. | 데이터 불변성 (Data Immutability), 감사 추적 가능성 (Audit Trail) |
| **1.2** | **데이터 출처 및 처리 과정 검증** | 외부 API를 통해 데이터를 수신하거나 변환하는 모든 로직에 대해, 데이터의 원본 소스(Source ID/Link)와 시스템 내에서 거친 *모든 가공 단계*를 로그로 남기도록 강제함. | ① **Provenance 기록:** '데이터 A는 [외부 DB X]로부터 오고, 게이트웨이에서 [필터링 로직 Y]를 거쳐 최종 저장됨'과 같은 흐름을 추적할 수 있어야 함. ② **처리 담당 모듈 명시:** 데이터를 변경한 코드 모듈 또는 함수명(e.g., `DataProcessorV2`)을 로그에 기록함. | 데이터 출처 투명성 (Data Provenance), GDPR Art. 5(1)(f) 준수 |
| **1.3** | **데이터 삭제 요청의 처리 (Right to Erasure)** | 사용자가 개인정보 삭제를 요청했을 때, 단순히 DB 레코드를 지우는 것이 아니라 '삭제되었음'을 증명하는 로그와 함께 *가명화(Pseudonymization)* 과정을 거치는지 검증함. | ① **Deletion Log 기록:** 누가, 언제, 어떤 필드의 데이터를 영구적으로 제거했는지의 감사 로그를 남기고, 해당 로그 자체도 접근 제어됨. ② **데이터 마스킹 확인:** 민감한 PII가 포함된 모든 결과물은 API 호출 시 즉시 '마스크 처리(예: XXXXX-****)'되어야 함. | GDPR Art. 17 (Right to Erasure), CCPA 준수 |

## Ⅱ. 보안 및 인증/권한 관리 강화 (Security & Compliance Depth)
단순히 `basic` 레벨의 토큰만 검사하는 것을 넘어, 산업별 규제 요구사항을 반영하여 다층적인 방어 메커니즘이 작동해야 합니다.

| No. | 검증 항목 (Verification Point) | 테스트 방법 (Test Case/Scenario) | 기대 결과 (Expected Compliance Output) | 관련 규제 준수 요소 |
| :---: | :---: | :---: | :---: | :---: |
| **2.1** | **MFA 미적용 시도 차단** | OAuth 토큰이 존재하지만, MFA(Multi-Factor Authentication)가 요구되는 민감 기능에 접근하려는 요청을 시뮬레이션함. | ① 게이트웨이가 해당 리소스의 정책 검증 단계에서 즉시 401 Unauthorized를 반환해야 하며, **'MFA 재인증 필요'**라는 명확한 에러 메시지를 반환해야 함. ② 이 실패 기록은 감사 로그에 남아야 함. | MFA 강제화 (Zero Trust Principle), PCI DSS 요구사항 |
| **2.2** | **최소 권한 원칙(PoLP)의 역추적 테스트** | `basic` 사용자 계정으로, 사실상 `admin`만 접근 가능한 특정 데이터베이스 컬럼에 직접적으로 쿼리 요청을 보내는 시도를 함. | ① 게이트웨이 레벨에서 요청 내용(Query Payload) 자체를 분석하여 권한 범위를 벗어나는 키워드나 필드가 발견되면, **실행 전** 트랜잭션을 차단하고 `Access Denied (PoLP Violation)` 에러를 반환해야 함. ② 데이터베이스 레이어의 접근 제어와 게이트웨이 레벨의 로직적 검증을 모두 수행함을 로그에 기록함. | PoLP(Principle of Least Privilege), 보안 아키텍처 분리 원칙 |
| **2.3** | **국가/지역별 데이터 전송 통제 (Data Residency)** | 사용자가 특정 지역(예: EU)의 데이터를 요청했으나, 내부 백엔드 서버가 다른 국가(예: 미국)에 위치하여 처리하는 것을 시도함. | ① 게이트웨이가 요청된 데이터의 '출발지'와 '처리할 위치'를 비교하고, 법적 제약이 있을 경우 트랜잭션을 거부해야 함. ② **데이터 전송 승인 기록:** 해당 데이터가 국경을 넘는 순간, 어떤 합법적 근거(SCCs 등) 하에 처리되었는지 로그로 남겨야 함. | GDPR Chapter V (International Data Transfer), 지역별 법규 준수 |

## Ⅲ. 운영 탄력성 및 예외 처리 (Resilience & Edge Case Handling)
시스템이 고장 나거나 비정상적인 부하를 받을 때, 그 실패 과정 자체가 규제 위반을 일으키지 않도록 설계되어야 합니다.

| No. | 검증 항목 (Verification Point) | 테스트 방법 (Test Case/Scenario) | 기대 결과 (Expected Compliance Output) | 관련 기술 목표 |
| :---: | :---: | :---: | :---: | :---: |
| **3.1** | **Rate Limiting 회피 시도 검증** | 봇(Bot)처럼 짧은 시간 내에 수백 개의 요청을 무작위 IP/헤더를 이용해 보내는 공격을 시뮬레이션함. | ① 단순히 차단하는 것을 넘어, 지연된 응답(Throttling Response)과 함께 **'현재 시스템 부하로 인해 처리가 지연되고 있음 (Rate Limit Exceeded)'**이라는 명확한 피드백을 제공해야 함. ② 공격 패턴 자체를 분석하여 비정상적인 요청 시도 횟수와 IP 대역을 즉시 경고 로그에 기록함. | DoS/DDoS 방어, 서비스 가용성(Availability) |
| **3.2** | **종속성 실패 처리 (Circuit Breaker)** | 백엔드의 핵심 데이터베이스 또는 외부 규제 API가 일시적으로 다운되거나 응답이 느려지는 상황을 강제로 발생시킴. | ① 게이트웨이가 즉각적인 에러 반환 대신, 사전에 정의된 대체 로직(Fallback Logic)이나 캐시 데이터를 사용하여 사용자에게 **"현재는 시스템 점검 중이며, 예상 복구 시간은 OOO입니다."**와 같이 권위적이고 정보를 제공하는 메시지를 보여줘야 함. ② 외부 종속성 실패 자체도 감사 로그에 기록되어 트러블슈팅의 근거가 되어야 함. | 시스템 탄력성(Resilience), 사용자 경험 유지 |
