## 📜 [기술 스크럼 플랜] 불변 감사 게이트웨이(IAG) 구축 계획서
**문서 목적:** IAG 핵심 API 개발 단계별 로드맵 및 QA 체크리스트 제공 (개발자용)
**담당 역할:** 코다리 (Developer), 리서처 (Researcher) 협업

### 🎯 목표 요약 (Goal)
시스템의 모든 중요 트랜잭션은 `IAG`를 거쳐야 하며, 다음 세 가지 원칙을 강제합니다.
1.  **법적 무결성:** 모든 데이터는 불변의 감사 로그(`AuditBlock`)로 기록되어야 합니다.
2.  **강력한 인증:** JWT 기반의 필수 토큰 검증을 통해 접근 통제가 이루어져야 합니다.
3.  **시스템 권위:** API 응답은 단순 성공/실패가 아닌, 법적 근거와 위기 진단 관점을 포함해야 합니다.

---
### 📂 1단계: 프로젝트 구조화 (Project Structuring)
개발 효율성과 유지보수성을 위해 모듈을 명확히 분리합니다.

*   **`core_gateway/models/`:** 모든 Pydantic 모델 정의 위치.
    *   ✅ `AuditBlock.py`: 표준 감사 기록 구조체 (시간, 트랜잭션 ID, 상태 코드, 법적 근거 등 필수 포함).
    *   ✅ `AuthToken.py`: JWT Claim 및 사용자 인증 정보 모델.
*   **`core_gateway/services/`:** 핵심 비즈니스 로직 및 서비스 계층 분리.
    *   ✅ `auth_service.py`: 토큰 생성, 검증, 만료 처리 로직 전담.
    *   ✅ `audit_service.py`: 감사 기록 작성(Write) 전문 모듈. (DB 연동/Mocking 포함).
*   **`core_gateway/api/`:** FastAPI 라우터 정의 및 엔드포인트 분리.
    *   ✅ `main_router.py`: 전체 API 진입점 역할을 담당하며, 요청 흐름 제어.

### 🔄 2단계: 필수 API 호출 순서 (API Flow Sequence)
모든 트랜잭션은 이 단계를 거쳐야 합니다. (Flow Diagram으로 이해시키기 적합합니다.)

**[Client Request] → [1. Gateway Entry] → [2. Authentication Check] → [3. Business Logic Execution] → [4. Audit Logging] → [5. Response]**

| 단계 | API 모듈/함수 | 설명 (Action) | 필수 검증 포인트 (Self-Correction/Testing) |
| :--- | :--- | :--- | :--- |
| **1. 진입점(Entry)** | `main_router.py` | 요청 수신 및 유효성 1차 검사. | - 헤더 누락 시 401 반환 여부 확인.<br>- 필수 파라미터 미제공 시 422 반환 여부 확인. |
| **2. 인증(Auth)** | `auth_service.py` | JWT 토큰 추출 및 유효성 검증 (만료, 서명 등). | - 만료된 토큰 처리 로직이 적절한 리다이렉트/에러 코드를 반환하는가? <br>- 권한(Scope) 기반 접근 제어가 완벽하게 작동하는가? |
| **3. 비즈니스 로직** | `business_service.py` (신규 모듈) | 핵심 기능 실행 및 트랜잭션 데이터 처리. | - 입력 데이터의 타입 안정성(Pydantic Validation)이 확보되는가? <br>- 예외 발생 시 롤백(Rollback) 처리가 완벽한가? |
| **4. 감사 기록** | `audit_service.py` | 트랜잭션 결과를 기반으로 불변 로그 생성 및 저장. | - *가장 중요:* API 호출 성공/실패 여부와 무관하게, 모든 시도가 반드시 Log 블록을 남기는가? <br>- 법적 근거(Legal Basis) 필드를 누락하지 않는가? |
| **5. 응답(Response)** | `main_router.py` | 클라이언트에 최종 결과를 `AuditBlock` 구조체와 함께 전달. | - 성공/실패 메시지가 단순 텍스트가 아닌, 규정된 포맷으로 제공되는가? <br>- 데이터 손상 없이 완전한 응답이 전송되는가? |

### ✅ 3단계: 단계별 검증 및 테스트 체크리스트 (QA Checklist)

| 영역 | 기능/시나리오 | 기대 결과 (Expected Result) | 실패 시 조치 사항 (Failure Action) | 담당 에이전트 |
| :--- | :--- | :--- | :--- | :--- |
| **보안** | 유효하지 않은 토큰 전송 | 401 Unauthorized + `AuditBlock`에 '인증 실패' 기록. | `auth_service`에서 즉시 트랜잭션을 중단시키고, 에러 로그를 상세히 남긴다. | 코다리 |
| **데이터 무결성** | DB 연결 끊김 (Simulated) | 503 Service Unavailable + 재시도 지침 포함된 응답. | 비즈니스 로직 실행 전 단계에서 시스템 오류 감지 및 사용자에게 명확한 메시지를 전달한다. | 코다리, 리서처 |
| **감사 기록** | 성공적인 트랜잭션 완료 | API 요청/응답 로그가 모두 `AuditBlock` 형태로 저장됨. | Log 기록 실패 시에도 API 응답을 차단하고 개발자에게 즉각 경고를 보낸다. (Fail-Safe) | 코다리, 리서처 |
| **프론트엔드** | Mini ROI 데이터 연동 테스트 | 시뮬레이션 결과가 법적/재무 리스크 관점에서 해석되어 보여야 함. | 1차 검증 단계에서 개발 완료된 스펙(Mini ROI)을 활용하여 End-to-End 테스트를 진행한다. | 레오, Designer |
