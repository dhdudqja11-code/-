## 1. 비즈니스 모델 및 전략 방향
* **제품군**: 초기 시장 테스트 후 '자체 촬영 및 커미션' 방식으로 전환 (저작권 리스크 최소화).
* **수익 모델**: 물리적 상품 판매가 아닌 '디지털 유틸리티 참여형 경험 티켓'으로 전환. 
* **가격 정책**: 단일 판매가 아닌 Tiered Pricing(예: $39/$99 번들 옵션) 모델 적용. 원가 기반이 아닌 고객 지불 의사(WTP) 측정 방식 채택.
* **핵심 성과 지표(KPI)**: Impact Tier Conversion Rate (ITCR), 저장률(Save Rate), 참여율(ER). 구매율보다는 WTP(지불 의사) 검증을 최우선으로 함.
* **시장 현지화**: 국내/일본 시장 진출. 콘텐츠 테마를 '한국적 감성(위로, 일상)'과 결합하고, 단순 풍경을 넘어 심리적 경험과 가치 부여에 초점.

## 2. 제품/서비스 설계 (UX/UI & 디자인)
* **디자인 시스템**: 모든 판매 페이지와 Mockup 제작 시 8pt 그리드 시스템 및 여백(Negative Space) 활용 원칙 필수 준수. 손글씨 느낌의 카피 배치.
* **브랜드 감성**: '황혼', '안개' 등 5가지 핵심 분위기. 따뜻한 톤과 필름 그레인 오버레이 적용.
* **구조**: 단순 자산 수집을 넘어, 모든 요소는 컴포넌트 기반 및 위젯 형태의 데이터 모듈로 처리. 
* **사용자 경험(UX)**: 구매(Purchase)가 아닌 참여 동기 부여에 초점을 둔 CTA 및 스토리텔링 결합.

## 3. 기술 및 운영 자동화 (개발/시스템)
* **시스템 아키텍처**: 수작업 방식에서 Python 기반의 자동화 파이프라인으로 전환. SKU ID 중심의 표준화된 API 엔드포인트 처리.
* **데이터 관리**: 모든 상품 정보는 중앙 관리 시스템(CMS) 기반 메타데이터 스키마로 구조화.
* **테스트 원칙**: 완벽한 설계보다 MVP 실행 및 A/B 테스트 검증 우선. 결제 플로우(PayPal) 연동 시, 프론트엔드가 아닌 서버 Webhook을 통한 E2E 통합 테스트 검증 필수.
* **로컬 AI 모델 운영**: 
  - 모든 에이전트는 모델 오케스트레이션 시 메모리 부족(OOM) 방지를 위해 더 작은 단일 모델(예: google/gemma-4-e4b)을 일괄 적용한다.
  - 안정성을 위해 Context Length는 8192~16384 수준으로 설정하며, 모델 로드 전 서버 실행 상태를 먼저 확인한다.

## 4. 지식 관리 및 시스템 엔진
* **지식 엔진 구축**: 단순 위키가 아닌 '지식 엔진(Knowledge Engine)'으로 설계. 도구 사용 의무화(Mandatory Tool Use).
* **문서 관리 원칙**: 의사결정 시 논의 과정과 분리하는 구조화 기준(Decision_Criteria_Guide) 적용. 
* **용량 최적화**: 프롬프트 초과 에러(Context Length Exceeded) 방지를 위해 용량이 큰 MD 파일은 주기적으로 핵심만 요약 압축하여 저장 공간을 관리한다.
